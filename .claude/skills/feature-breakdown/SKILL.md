# Feature Breakdown Skill

## Overview

新機能の実装タスクを洗い出し、依存関係を考慮した順序で一覧提示する。

「何をどの順番で実装するか」を明確にすることが目的。

---

## Instructions

### Step 1: 機能の概要を把握する

実装する機能の以下を確認する：

- 何をするか（ユーザー操作・データ変化）
- DBスキーマの変更が必要か
- 新エンドポイントが必要か
- 新画面・新コンポーネントが必要か

---

### Step 2: タスクを洗い出す

以下のレイヤーごとにタスクを列挙する。

#### バックエンド
- DBスキーマ変更（Drizzle）
- マイグレーション生成（`db:generate`）
- ドメイン型定義
- Repository（DB操作）
- Service（ビジネスロジック）
- バリデーション（Zod）
- Controller（リクエスト受け取り・レスポンス返却）
- Router へのエンドポイント登録
- テスト

#### フロントエンド
- RPC 呼び出し hooks
- Presentational コンポーネント
- Container コンポーネント（hooks との接続）
- ルート定義追加
- ナビゲーション・メニュー更新
- テスト

---

### Step 3: 依存関係を整理して順序を決める

以下の原則で順序を決める：

1. **DBスキーマ → バックエンド → フロントエンド** の順が基本
2. バックエンド内では **Repository → Service → Controller → Router** の順
3. フロントエンド内では **hooks → Presentational → Container** の順
4. テストは各レイヤーの実装直後に行う
5. フロントエンドはバックエンドの型が確定してから着手する（RPC 型推論のため）

---

### Step 4: 結果を出力する

examples/expected-output.md の形式で提示する。

---

## Constraints

- 実装は行わない（計画作成のみ）
- 不要なタスクは含めない（機能に関係するものだけ列挙する）
- 各タスクには依存する前提タスクを明記する
