# Expected Output — 設計案提示

## 入力例

「タスクの削除を論理削除にしたい。どう設計すればいい？」

---

## 期待される出力

```
## 設計案

### 案A: deleted_at カラムを追加する（ソフトデリート）

`tasks` テーブルに `deleted_at TIMESTAMP` カラムを追加し、
削除時に現在時刻をセット、復元時に NULL に戻す。

**メリット**
- 実装がシンプル
- 復元が容易（NULL に戻すだけ）
- 削除日時が記録される

**デメリット**
- 全クエリに `WHERE deleted_at IS NULL` の条件追加が必要
- 条件の書き忘れによるバグリスクがある

**実装コスト**: 低

---

### 案B: is_deleted フラグを追加する

`deleted_at` の代わりに `is_deleted BOOLEAN` を追加する。

**メリット**
- 意味が明示的でわかりやすい

**デメリット**
- 削除日時が記録できない
- 案Aと比べて情報量が少ない

**実装コスト**: 低

---

### 案C: 削除済みテーブルを分離する

`deleted_tasks` テーブルを別途作成し、削除時に移動・復元時に戻す。

**メリット**
- メインテーブルを汚染しない
- 削除済みデータのクエリが分離される

**デメリット**
- 実装が複雑（移動処理・外部キー整合性の管理）
- マイグレーションコストが高い

**実装コスト**: 高

---

## 推奨案: 案A（deleted_at カラム追加）

**理由**: 実装コストが最も低く、削除日時の記録・復元の両方に対応できる。
Drizzle ORM でのフィルタ条件追加も容易。

## 破綻するケース

- 削除済みデータを別のストレージ（アーカイブDB等）に移したい要件が出た場合、案Cへの移行コストが発生する
- 削除件数が膨大になりクエリパフォーマンスが問題になった場合、インデックス設計の見直しが必要になる
```
